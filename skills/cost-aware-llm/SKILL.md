# Cost-Aware LLM Pipeline — コスト意識モデルルーティング

## 概要

全てのタスクにOpusを使う必要はない。
タスクの複雑さに応じてhaiku/sonnet/opusを使い分け、コストを最適化しつつ品質を維持する。

## モデル選択ガイド

### Haiku（最安・最速）— 単純タスク用

| 用途 | 例 |
|------|-----|
| ファイル検索・パターンマッチ | `Explore`サブエージェント |
| 単純なコード変換 | 変数名リネーム、import整理 |
| 分類・振り分け | エラー種別の判定 |
| フォーマット変換 | JSON↔YAML、Markdown整形 |

**Agentツール指定**: `model: "haiku"`

### Sonnet（バランス）— 標準タスク用

| 用途 | 例 |
|------|-----|
| コードレビュー | 個別ファイルのレビュー |
| 実装（中規模） | 1-3ファイルの変更 |
| テスト生成 | ユニットテスト作成 |
| ドキュメント生成 | JSDoc、README更新 |

**Agentツール指定**: `model: "sonnet"`

### Opus（最高品質）— 複雑タスク用

| 用途 | 例 |
|------|-----|
| アーキテクチャ設計 | 新システムの設計判断 |
| 深いデバッグ | 複数ファイルにまたがるバグ |
| セキュリティ分析 | 脆弱性の検出と修正 |
| 複雑なリファクタリング | 10+ファイルの構造変更 |

**Agentツール指定**: `model: "opus"` または指定なし（デフォルト）

## 実践的ルーティング

### サブエージェントレビュー時

```
Tier 1 レビューアー（arch/security/perf）→ sonnet
  理由: 定形的なチェックリストベースの検査

Tier 2 レビューアー（test/code-quality等）→ sonnet
  理由: パターンマッチング主体

Explore サブエージェント → haiku
  理由: 検索・読み取りのみ

learnings-researcher → haiku
  理由: grep + スコアリングのみ
```

### 実装時

```
単一ファイル修正 → sonnet
複数ファイル連携修正 → opus
テスト生成 → sonnet
設計判断を含む実装 → opus
```

## コスト削減のTips

1. **`MAX_THINKING_TOKENS=10000`**: 思考トークンを制限（設定済み）
2. **`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50`**: 早めのcompact（設定済み）
3. **サブエージェントのmodel指定**: 上記ガイドに従う
4. **不要なコンテキスト排除**: `/compact`を論理的なブレークポイントで実行
5. **並列 > 逐次**: 同じ情報を何度も渡すより並列で1回ずつ

## 月間コスト目安

| パターン | 推定コスト比 |
|----------|------------|
| 全てOpus | 100% |
| Opus + Sonnetサブエージェント | ~60% |
| Opus + Sonnet + Haikuサブエージェント | ~40% |
